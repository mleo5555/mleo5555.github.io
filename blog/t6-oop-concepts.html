Classes vs. Modules

8/16/15

Classes and modules share similarities in Ruby but are utilized in very different ways - let's compare.

Classes

Classes are instantiated and create a new object within Ruby. A class consists of methods, constants, and variables. Think of a class as the blueprints required to create a new object. Let's use an example:

class Car

   def initialize(brand, name)

      @brand = brand

      @name = name

   end

   def vroom

      puts 'Vroooom!'

   end

   def details

      puts "This fine car is a #{@brand} #{@name}."

   end

end

Within the class Car are 3 methods: initialize, vroom, and details. All three methods create the behaviors of the class Car. In order to use a class, you must instantiate it with the following:

my_car = Car.new(Ford, Mustang)

This code creates a new instance of the class Car called my_car and assigns (Ford, Mustang) to the instance variables (@brand, @name). Since my_car is an object, methods can be called upon it. For example:

my_car.details will puts This fine car is a Ford Mustang.

Modules

Unlike classes, modules cannot be instantiated. Think of modules as libraries that hold bundles of methods, constants, and classes. Let's look at the Ruby Math module as an example:

module Math

   PI = 3.141592654

end

There are other constants and methods within this module but we will keep it simple for now and only refer to the constant PI. Since we cannot instantiate a module, we must extract its data differently than we would for a class. Here are two simple ways access PI:

puts Math::PI

#=> 3.141592653589793

include Math

puts PI

#=> 3.141592653589793

The first example uses the constant lookup operator (::). This allows you to scope constants and classes within a module. The second example uses include which allows the mix in of modules into classes.
